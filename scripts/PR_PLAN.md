# PR plan: `@mastra/voice-inworld-realtime`

**Goal:** Add Inworld Realtime API as a first-class full-duplex voice provider in `mastra-ai/mastra`, alongside `@mastra/voice-openai-realtime`, `@mastra/voice-google-gemini-live`, and `@mastra/voice-aws-nova-sonic`. Inworld already ships as a non-realtime TTS voice provider (`@mastra/voice-inworld`) — this is the realtime/speech-to-speech sibling.

**Why this is a small PR:** Inworld's realtime websocket protocol is essentially the OpenAI Realtime **GA spec** with one rename. Verified by raw-websocket smoke tests in this directory (see `inworld-realtime-smoke.mjs` and `inworld-realtime-smoke-text.mjs`). The Inworld public docs at https://docs.inworld.ai/api-reference/realtimeAPI/realtime/realtime-websocket are partially stale — **trust the smoke-test output, not the docs**, where they conflict.

---

## Verified protocol facts (from live API, not docs)

**Connection**
- URL: `wss://api.inworld.ai/api/v1/realtime/session`
- Auth: `Authorization: Basic ${INWORLD_API_KEY}` — key is **already Basic-encoded**, pass verbatim. Do **not** base64-re-encode.
- Audio default: PCM16 @ 24kHz. Also supports `audio/pcmu` and `audio/pcma` @ 8kHz.

**Client → server events (all OpenAI-compatible, identical names)**
- `session.update`, `session.update` keys mirror OpenAI's
- `conversation.item.{create,delete,retrieve,truncate}`
- `response.{create,cancel}`
- `input_audio_buffer.{append,commit,clear}`
- `output_audio_buffer.clear`

**Server → client events observed live (OpenAI GA spec)**
```
session.created
session.updated
conversation.item.added          <-- ONLY rename vs OpenAI ("created" → "added")
conversation.item.done           <-- Inworld extension, per-item completion
response.created
response.output_item.added
response.content_part.added
response.output_audio.delta
response.output_audio_transcript.delta
response.output_text.done
response.content_part.done
response.output_audio_transcript.done
response.output_audio.done
response.output_item.done
response.done
response.function_call_arguments.delta    <-- singular, OpenAI-compatible (docs say plural; docs are wrong)
response.function_call_arguments.done
error
```

**Documented but not observed (may still fire under VAD-driven scenarios):**
- `input_audio_buffer.speech_started` / `speech_stopped` (need a mic-streaming test to trigger)

**Stale doc claims contradicted by the live API:**
- ❌ Docs: "session.created not currently supported" → ✅ Live: emitted on every connect
- ❌ Docs: `response.started` → ✅ Live: `response.created`
- ❌ Docs: `response.function_calls_arguments.delta` (plural) → ✅ Live: `response.function_call_arguments.delta` (singular)
- ❌ Docs: `response.output_item.delta` → ✅ Live: `response.output_item.added` + per-content-part deltas

**Inworld-specific extensions:**
- `providerData` on `session.update` — passthrough for Inworld-only knobs (voice presets like `Dennis`, TTS model selection, playback speed 0.25–1.5x, semantic VAD eagerness, MCP tool_choice).
- Voice presets differ from OpenAI's. Get current list from https://api.inworld.ai voices endpoint or hardcode the documented set (`Dennis`, ...). Recommend a `getSpeakers()` that fetches dynamically.
- `tool_choice: "mcp"` is an Inworld addition.

## Delta vs `@mastra/voice-openai-realtime`

Only three things differ at the protocol level:

1. **URL + auth scheme**: different base URL; `Basic` not `Bearer`.
2. **One event rename**: `conversation.item.added` (Inworld) ↔ `conversation.item.created` (OpenAI).
3. **One extra event**: `conversation.item.done` (Inworld emits per-item, OpenAI doesn't).

Everything else (`response.output_audio.delta`, `response.output_item.added`, `response.content_part.added`, function-call deltas, `response.done`, `session.created/updated`, error, etc.) is **byte-identical event names with compatible field shapes**.

### 4. `src/index.ts` — `InworldRealtimeVoice` class

Port `OpenAIRealtimeVoice` with these specific changes:

| What | OpenAI version | Inworld version |
|---|---|---|
| `DEFAULT_URL` | `wss://api.openai.com/v1/realtime` | `wss://api.inworld.ai/api/v1/realtime/session` |
| `DEFAULT_MODEL` | `gpt-4o-mini-realtime-preview-2024-12-17` | `anthropic/claude-sonnet-4-6` (or whichever model Inworld recommends for realtime) |
| `DEFAULT_VOICE` | `alloy` | `Dennis` (or fetch dynamically) |
| `VOICES` | static 8-voice array | fetch from Inworld voices API at construct or on `getSpeakers()` |
| Auth header | `Authorization: Bearer ${apiKey}` | `Authorization: Basic ${apiKey}` (verbatim, NOT re-encoded) |
| `apiKey` env fallback | `process.env.OPENAI_API_KEY` | `process.env.INWORLD_API_KEY` |
| Handshake | wait for `session.created` then `session.updated` | wait for `session.updated` (session.created arrives but isn't strictly required; treat both as valid signals) |
| Conversation item event | `conversation.item.created` | `conversation.item.added` |
| Per-item completion | none | `conversation.item.done` (Inworld extension — log/ignore or surface) |

Add a new constructor option:
```ts
constructor(options: {
  model?: string;
  url?: string;
  apiKey?: string;
  speaker?: string;
  debug?: boolean;
  providerData?: Record<string, unknown>;  // merged into every session.update
} = {}) { ... }
```

`providerData` should be **shallow-merged** into the `session` object on every `session.update` we send. This is the Inworld extension hook.

**Critical bug-prevention check:** OpenAI's preview API used `response.audio.delta` / `response.audio_transcript.delta`. The GA spec (which Inworld matches) uses `response.output_audio.delta` / `response.output_audio_transcript.delta`. Make sure the event-name strings in the receive handler are the GA versions — not the preview names. If you copy-paste from the OpenAI Mastra package and it's listening for preview names, you'll silently drop all audio in the Inworld version.

### 6. Tests (`src/index.test.ts`, `src/utils.test.ts`)

Mirror the OpenAI package's test structure. Use the `ws` library's `WebSocketServer` to mock an Inworld server. Coverage targets:

- ✅ Connect sends Basic auth header (not Bearer) with the key verbatim

- ✅ `session.created` then `session.updated` both received without error
- ✅ `speak()` flow: `conversation.item.create` → `response.create` → emits `speaker` events on `response.output_audio.delta`
- ✅ `listen()` flow: streaming PCM in → `input_audio_buffer.append` chunks → emits `writing` events on transcription deltas
- ✅ Function/tool call: `response.function_call_arguments.delta` (singular) parsed correctly; full JSON assembled on `.done`
- ✅ `conversation.item.added` correctly routed (would break if copying OpenAI's `conversation.item.created` handler)
- ✅ Interruption: `output_audio_buffer.clear` cancels current playback
- ✅ `providerData` constructor option is merged into outgoing `session.update`
- ✅ Reconnect / close cleanup

## Out of scope (deliberate, to keep the PR reviewable)

- WebRTC transport — Mastra's OpenAI realtime is WS-only; match that.
- First-class API for `tool_choice: "mcp"` — surface via `providerData` only.
- Voice cloning / custom voices — same: `providerData`.
- Server-VAD vs semantic-VAD config UI — passthrough via the standard `audio.input` shape; Inworld extensions go in `providerData`.

## Open questions for the implementing agent

1. **Voice listing** — does Inworld expose a `GET /v1/voices` (or similar) for realtime? If yes, `getSpeakers()` should fetch dynamically. If no, hardcode the documented set and re-fetch periodically.
2. **Realtime model identifiers** — confirmed `anthropic/claude-sonnet-4-6` works (smoke test passed). Are there Inworld-native realtime-only models we should default to instead? Ask the Inworld team.
3. **`response.output_text.delta`** — was not observed in the text-output smoke test (only `response.output_text.done` was emitted, with the full text in the audio_transcript stream). Check whether text-only modality (no audio) emits `response.output_text.delta`. If so, the handler needs to support both delta and done.
4. **VAD events** — smoke tests didn't trigger `input_audio_buffer.speech_started/stopped` because we sent only text. A third smoke test that streams a PCM file into `input_audio_buffer.append` would confirm these event names match the docs.
